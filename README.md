## 소개

서비스가 단순히 동작하는 것보다, 문제가 생겼을 때 원인을 따라갈 수 있는 구조에 더 관심이 많습니다.

요청이 어디에서 시작되고, 어떤 기준으로 처리되며, 실패했을 때 어떤 기록이 남는지를 중요하게 봅니다.

## 관심 있는 문제

| 영역 | 보는 기준 |
| --- | --- |
| 서비스 흐름 | 요청이 어디에서 시작되고 어디에서 멈추는지 |
| 장애 분석 | 실패를 다시 확인하고 재처리할 수 있는지 |
| 운영 가시성 | 로그, 지표, 추적이 다음 행동으로 이어지는지 |
| 자동화 | 반복을 줄이고 실수를 줄이는지 |
| 개발 도구 | 흐름을 방해하지 않고 판단을 돕는지 |

## 작업할 때 보는 것

- 먼저 흐름을 나눕니다.
- 추측보다 로그와 재현을 봅니다.
- 실패했을 때 남는 정보를 중요하게 봅니다.
- 오래 유지될 수 있는 단순한 구조를 선호합니다.

## 기록

겪었던 문제와 해결 과정을 블로그에 남깁니다.

주로 원인 추적, 설계 판단, 운영 중 드러난 차이, 나중에 다시 봐야 할 결정들을 정리합니다.
